# 2026-03-30 工作日誌：Qwen3-Coder Tool-Call 增強計畫

## 今日目標

在 Intel Core Ultra iGPU 上改善 Qwen3-Coder-30B-A3B-Instruct INT4 模型的**長上下文工具呼叫準確度**，目標是 24K-32K token 下保持穩定的 tool-call 能力。

---

## 背景摘要

### 問題

使用 INT4 量化的 Qwen3-Coder-30B-A3B 作為 coding agent 時，在 **24K+ token** 的 hard agentic 場景下工具呼叫會退化：
- 模型不再執行被要求的工具呼叫
- 改為「敘述分析」對話歷史中的程式碼
- 這是 **注意力層 INT4 量化** 造成的，NOT 路由器問題

### 已完成的實驗

| 實驗 | 結果 |
|---|---|
| 基線 INT4 基準測試 | NIAH 100%、簡單 tool-call 100%、**hard tool-call 在 24K+ 退化** |
| 路由器反量化 (INT4→FP16) | **無顯著改善** — 證實路由器不是瓶頸 |
| Gemini 分析審查 | 多數建議錯誤；唯一有價值的：長上下文校準資料 |

### 基線 Hard Tool-Call 分數

| 上下文長度 | 4K | 8K | 16K | 24K | 32K |
|---|---|---|---|---|---|
| INT4 基線 | 4.8/5 | 4.8/5 | **5.0/5** | 4.0/5 | 4.2/5 |

### 擴展壓力測試 (2026-04-02, 32K-64K)

| 測試 | 32K | 48K | 64K |
|---|---|---|---|
| 簡單 Tool-Call (7 tools) | **5.0/5** | **5.0/5** | **5.0/5** |
| **Hard Tool-Call (12 tools)** | **4.0/5** | **4.2/5** | **4.0/5** |

失敗詳情：
- `read_file`: 32K=FAIL(1/5, 70s), 48K=FAIL(1/5, 155s), 64K=FAIL(0/5, 218s)
- `replace_in_file`: 32K=PARTIAL(4/5) — 選錯工具用了 `read_file`
- 失敗原因：歷史中有~95次相同的 `read_file(handlers.py)` 呼叫，模型認為已讀過而拒絕執行

### 關鍵發現

1. **NIAH 和簡單 tool-call 不受 INT4 影響** — 到 **64K** 都完美
2. **Hard agentic tool-call 在 24K+ 退化** — 模型無法在大量對話歷史中優先處理最新指令
3. **MoE 路由器量化不是原因** — 48 個路由器反量化到 FP16 毫無效果
4. **注意力層是瓶頸** — 192 個 INT4 q/k/v/o_proj 層影響長上下文注意力模式
5. **Intel Arc B390 GPU 只支援 uniform INT4 gs=128** — ratio<1.0(混合INT4/INT8)崩潰, gs=64崩潰
6. **AWQ 在 MoE 模型上壞掉** — NNCF 2.19.0 `matmul shape mismatch`
7. **NNCF 校準在 ~1400 tokens 以上 OOM** — 96GB RAM 不夠用
8. **LayerNorm 權重已是 FP32** — 不需要保護
9. **embed_tokens/lm_head 已是 INT8** — 基線轉換已經保護了這些
10. **長上下文校準資料是關鍵缺口** — 標準短文本校準無法教 scale_estimation 長上下文注意力模式

## 今日計畫

### 第一步：GPU 相容性實驗 [已完成]

系統性測試 Intel Arc B390 iGPU 支援的 INT4 參數組合：

| 組合 | GPU 結果 |
|---|---|
| gs=128, ratio=1.0 | **OK** |
| gs=128, ratio=1.0, RouterFP16 | **OK** |
| gs=128, ratio=0.9 | **崩潰** (segfault) |
| gs=128, ratio=0.8 | **崩潰** (segfault) |
| gs=64, ratio=1.0 | **崩潰** (segfault) |
| gs=64, ratio=0.8 | **崩潰** (segfault) |

**結論：** 只有 uniform INT4 gs=128 ratio=1.0 能在 GPU 上運行。

### 第二步：NNCF 記憶體限制測試 [已完成]

| MAX_SEQ_LEN | 結果 |
|---|---|
| 1024 tokens | OK（但只有截斷的 system prompt，沒有 tool-call 資料）|
| 1280 tokens | OK（精簡 prompt，0/8 樣本被截斷）|
| 1536 tokens | OOM（1.43 GB 分配失敗）|
| 2048 tokens | OOM（768 MB 分配失敗）|

**結論：** 硬限制 ~1300-1400 tokens/sample。

### 第三步：convert_hq.py 修復 [已完成]

- AWQ 預設關閉（MoE 崩潰），加 `--awq` flag
- 精簡 system prompt（389 tokens vs 1865 tokens）
- MAX_SEQ_LEN = 1280
- EXCHANGE_RANGE = (1, 2)
- 預截斷診斷：顯示 token 數、截斷旗標
- 儲存校準 prompts 到磁碟供檢查

### 下一步：NNCF 校準轉換 [待執行]

**可行方案：** ratio=1.0 + scale_estimation + tool-calling 校準資料

```bash
python scripts/convert_hq.py --tier tier1 --dataset long_tool_calling \
  --group-size 128 --ratio 1.0 --subset-size 32 \
  --sensitivity-metric max_activation_variance \
  --output-dir "INT4-TOOLCAL"
```

**預期效果：** scale_estimation 使用 tool-call 資料最佳化 INT4 scales/zero-points，
可能改善 hard tool-call 在 24K+ 的表現（基線 4.0/5 → 目標 4.6+/5）。

**已排除的方案：**
- AWQ: MoE 崩潰
- gs=64: GPU 崩潰
- ratio<1.0: GPU 崩潰（混合 INT4/INT8 不支援）
- MAX_SEQ_LEN>1400: OOM

## NNCF 可用的敏感度指標

| 指標 | 說明 | 適用場景 |
|---|---|---|
| `max_activation_variance` | 最大激活值方差 | **預設首選** — 識別激活值波動最大的層 |
| `mean_activation_variance` | 平均激活值方差 | 較穩定但可能遺漏極端情況 |
| `mean_activation_magnitude` | 平均激活值幅度 | 適合注意力層 |
| `hessian_input_activation` | 基於 Hessian 的輸入激活分析 | **最精確但最慢** — 計算每個權重對損失的影響 |
| `weight_quantization_error` | 純權重量化誤差 | 不需要校準資料，最快但最粗糙 |

## Gemini 分析審查摘要

| Gemini 建議 | 判定 | 我們的證據 |
|---|---|---|
| 保護 `gate_proj` | **錯誤** — 搞混 SwiGLU 門控和 MoE 路由器 | 路由器是 `mlp.gate`，不是 `gate_proj` |
| 保護 LayerNorm | **不必要** | 已驗證：LayerNorm 權重是 FP32 [1,1,2048] |
| 保護 embed/lm_head | **已做到** | 基線已是 INT8 |
| `hessian_trace` 指標 | **不存在** | NNCF 有 `HESSIAN_INPUT_ACTIVATION`（不同） |
| Group size 64 | ~~有效~~ **GPU 不支援** | Intel Arc B390 segfault on gs=64 |
| AWQ | ~~有效~~ **MoE 崩潰** | NNCF 2.19.0 `matmul shape mismatch` |
| scale_estimation | **有效** ✅ | 主要的品質提升手段（不需 AWQ） |
| 20K+ 校準資料 | **非常有價值** ✅ | **已採納** — 新增 `long_tool_calling` 資料集 |
| Ratio 0.85 | ~~值得測試~~ **GPU 不支援** | ratio<1.0 = 混合 INT4/INT8 → GPU segfault |
